- 单页快速检测：tab计数≤20时走快速通道，不进循环

## 性能优化记录（2026-05-29）

### ✅ 串行→并发审查
- **修改：** `run_compliance_claim.py` 第237行，`checker.review_batch(products)` → `checker.review_batch_concurrent(products, max_workers=5, ...)`
- **预期提速：** 5个商品审查从 ~5×3s=15s 降至 ~3s（5线程并发）
- **附带：** 新增实时进度回调 `📊 审查进度: done/total`

### ✅ 分层 Vision 策略
- **修改：** `compliance_checker.py` `_review_one()` + `_need_vision_check()` + `image_checker.py` `check(vision_mode=)`
- **逻辑：**
  1. 先做 OCR only（`vision_mode="ocr_only"`）
  2. OCR 发现违规 → 直接 reject（省掉 Vision 调用）
  3. OCR 通过 + 低风险类目（五金/3C/百货/户外/宠物/家居） → **跳过 Vision**
  4. OCR 通过 + 高风险类目（服装/母婴/食品/美妆） → 调 Vision（`vision_mode="always"`）
  5. OCR 置信度不足 (<0.8) → 无论类目都走 Vision
- **配置：** `config/category_list.json` 新增 `vision_risk` 字段，可灵活调整每个类目的风险等级

### 性能瓶颈分析
之前工作流执行慢的主要原因：
1. **串行审查**：每个商品依次调 LLM API，5个商品需 15-25s（已修复→并发）
2. **每次都调 Vision**：低风险商品也走 Vision API，每个 ~3-8s（已修复→分层跳过）
3. **滚动勾选耗时**：P0 v2 策略B 需 400+ 步细滚动，每步 120ms 等待，总计 ~50s（优化空间有限，虚拟滚动限制）
4. **OCR 因 Tesseract 缺失全部失败** → 退化为全走 Vision（已安装 Tesseract 后可缓解）

## 核心业务逻辑

### 过审未分配商品 = 留在采集箱
- **审查通过**但**没匹配到店铺类目**的商品 → 留在采集箱，不认领、不删除
- 只有**审查不通过**（reject）的商品才会被删除
- 流程：审查(review-only) → 分配类目 → 只认领有匹配的(direct-claim) → 发布

### 删除策略
- `reject_ids`（审查不合规） → 自动从采集箱删除
- `unassigned`（过审无类目） → 保留在采集箱
- 绝对不会误删过审商品
