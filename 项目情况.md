## 已修复的问题

### ✅ LLM 标题审查修复（2026-05-30 关键修复）
**症状：** `LLMClient.chat(messages, response_format={"type": "json_object"})` 抛出 `got an unexpected keyword argument 'response_format'`，LLM 审查静默失败，所有商品全 pass（含宝可梦等违规品）。异常被 `except` 吃掉，无报错提示。
**根因：** `LLMClient.chat()` 不支持 `response_format` 参数，错误是原代码就有的。
**修复：** 改用 `client.chat_json()` 方法，该方法内部调用 `chat()` 后手动解析 JSON，且有 `_parse_json_response()` 兜底处理格式异常。
**影响范围：** 之前所有的标题审查结果都不正确。修复后正确识别出宝可梦×3、凯蒂猫×2、星之卡比×1 等 IP 侵权商品。
**涉及代码：** `infrastructure/compliance_checker.py` `_review_one()`

### ✅ 删除功能批量 JS 滚动修复（2026-05-30）
**症状：** `delete_rejected_products()` 报告删除 7 件，实际只删 1 件。原因是 Vue Recycle Scroller 未渲染中间行。
**根因：** `erp_publisher.py` 的滚动使用了 `page.evaluate("for(...){window.scrollTo(0, y)}")` — 同步 JS 循环中所有中间 scroll 位置被浏览器合并，只渲染最终位置。
**修复：** 改为 Python 循环逐格 scrollTo + sleep(80ms)，每一步都给浏览器时间渲染。
**涉及代码：** `infrastructure/erp_publisher.py` delete_rejected_products()

### ✅ Claim-and-Replace 循环策略（2026-05-30）
**替代：** 废弃原策略A/B/C（约400行代码），改为仅扫第1页可视行→匹配ID→勾选→认领→已认领商品从tab消失→后续页自动填充到第1页→循环。
**优势：** 不受分页数和虚拟滚动限制，代码精简到~80行。
**涉及代码：** `run_compliance_claim.py` `[4/5]` 勾选+认领逻辑 + `run_publish.py` target_ids 路径

### ✅ 过审未分配商品留在采集箱（2026-05-30 业务逻辑）
**症状：** 过审但没匹配到类目的商品被一起认领到店铺（如羽毛球→户外类，吉象星連坊无映射）
**修复：** 流程重构为 审查(review-only)→分配类目→只认领有匹配的(direct-claim)→发布
**涉及代码：** `run_compliance_claim.py` + `run_workflow.py`

### ✅ 删除逻辑重写（2026-05-30）
**旧逻辑：** 遍历所有分页后一次性勾选→删除（虚拟滚动跨页勾选不可靠）
**新逻辑：** 循环法——只扫第1页→勾选匹配 → 删除 → 确认 → 刷新 → 重复
**涉及代码：** `infrastructure/erp_publisher.py` delete_rejected_products()

## 性能优化记录

### ✅ sleep 全面压缩（95s → ~33s）
- `TWO_SECONDS(19处)→800ms`, `THREE_SECONDS(8处)→1000ms`, `ONE_SECOND(5处)→400ms`, `HALF_SECOND(16处)→200ms`
- `PUBLISH_INITIAL(5000ms)→2000ms`, `SCROLL_FAST(150ms)→80ms`

### ✅ 合并 LLM 调用（2次/商品 → 1次）
标题审查 + 类目识别合并为单次 LLM 调用，节省 ~50% LLM API 时间。

### ✅ 去除冗余导航 + 批量 JS 滚动（已回滚）
- 去除了 claim 阶段的重复 navigate_to_collection_box + 关弹窗 + 切 tab
