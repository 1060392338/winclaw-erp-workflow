# 第一篇：基础认知 — AI怎么帮你写代码

## 1.1 这套方法的核心思想

**人类负责「规划」，AI 负责「执行」。**

```
人类：我知道要做什么（业务知识）     ← 你有
AI：我知道怎么写代码（技术实现）       ← AI 有
```

就像你不会建房子，但你知道自己要「一个三室一厅有阳台的公寓」——你画出草图，施工队（AI）帮你建。

## 1.2 工作流是什么

**工作流 = 把多个步骤自动串起来执行。**

就像工厂流水线：
```
原料 → 步骤1 → 步骤2 → 步骤3 → 成品
```

在我们的项目中：
```
采集箱商品 → 合规审查 → 删除不合格 → 认领到店铺 → 发布到Shopee
```

**用手做 vs 用代码做：**

| 手动操作 | 代码操作 |
|---------|---------|
| 打开浏览器 → 点采集箱 → 看商品图片 → 判断是否合规 → 勾选 → 点认领 → 选店铺 → 去发布页 → 点发布 | `python run_workflow.py --claim-to "順順の小屋童裝"` |

**好处：** 一次写好，每次点一下跑完。不会漏步骤、不会点错、不用盯着。

## 1.3 你要懂多少代码？

**你不需要：**
- ❌ 不需要懂编程理论
- ❌ 不需要背语法
- ❌ 不需要从零写代码

**但你需要知道：**
- ✅ 怎么打字（会打字就会）
- ✅ 怎么打开文件（会双击就行）
- ✅ 怎么复制粘贴（Ctrl+C / Ctrl+V）
- ✅ 怎么安装软件（下一步 → 下一步 → 完成）

**可以这样理解：你把AI当成一个什么都会但需要你指挥的程序员，你只需要告诉他干什么，他负责写出代码。**

## 1.4 三个核心原则

| 原则 | 解释 |
|------|------|
| **先写规划，再写代码** | 不要上来就写代码，先和AI一起定好要做什么 |
| **拆成小块，逐块推进** | 一次只让AI干一件事，做完验收再下一块 |
| **AI写的你要验证** | AI也会犯错，跑一遍就知道对不对。不要盲信 |

# 第二篇：工作流规划 — 先想清楚再动手

## 2.1 画出你的流程

在写任何代码之前，先在纸上（或脑里）画出你的流程。

### 问自己三个问题

| 问题 | 例子（本项目） |
|------|--------------|
| ① 我的起点是什么？ | 采集箱里有未处理的商品 |
| ② 我要做什么操作？ | 审查商品是否合规 → 认领到店铺 → 发布 |
| ③ 最后要什么结果？ | 商品在Shopee上架成功 |

### 画流程图

不要用复杂工具，画箭头就行：

```
采集箱商品 → [AI审查] → 合规？→ 是 → 认领 → 发布 → 完成
                           ↓ 否
                         删除
```

**画完之后问自己：哪些步骤可以用代码代替手动操作？答案是：几乎全部。**

## 2.2 拆成小步骤

**大任务拆成小步骤，每个小步骤只做一件事。**

### 不要这样写

```
python_做所有事情.py   ← ❌ 一个文件几千行，改一处全崩
```

### 要这样写

```
项目文件夹/
├── step1_collect.py      # 采集商品
├── step2_review.py       # 审查是否合规
├── step3_claim.py        # 认领到店铺
├── step4_publish.py      # 发布到 Shopee
├── run_all.py            # 总控：按顺序调用上面4个
├── .env                  # API密钥（不上传）
└── README.md             # 说明文档
```

**每个脚本只负责一件事**，好处：
- 哪个步骤出问题，只改那个文件
- 可以单独测试某个步骤（不用每次都跑全部）
- 以后想调整顺序，改总控就行

## 2.3 写规划文档（喂给AI的"设计图"）

**规划文档 = 你告诉AI你要做什么，AI基于这个文档一步步写代码。**

### 规划文档模板

```markdown
# 项目：[项目名称]

## 目标
[一句话描述]

## 流程
[画流程图或步骤]

## 每个步骤做什么
1. [步骤1描述]
2. [步骤2描述]
3. [步骤3描述]

## 技术约束
- 操作系统：
- Python版本：
- 用到的工具/库：
- 浏览器连接方式：
- API密钥管理方式：
- 特殊注意事项：
```

### 怎么喂给AI

**第一次对话，把规划文档发给AI：**

```
你好，我要开发一个[项目名]。这是我的规划文档：

[粘贴完整规划文档]

我们先从项目整体看看，有什么需要补充或调整的吗？
```

### 写规划文档的技巧

如果AI给出来的方案有多个选择：

```
有两个方案——
A：[方案A]
B：[方案B]
你觉得哪个好？为什么？
```

**假装你在给一个人解释**——他不懂你的业务，但他会写代码。说清楚**输入是什么→要做什么→输出是什么**就行。

# 第三篇：推进执行 — 让AI逐块产出代码

## 3.1 把规划拆成开发清单

把规划文档变成一个代办清单：

```markdown
## 阶段1：基础设施（约30分钟）
- [ ] 项目文件夹结构搭建
- [ ] 环境检查和连接测试
- [ ] .env 配置文件模板

## 阶段2：[模块A]（约1小时）
- [ ] 步骤A1
- [ ] 步骤A2
- [ ] 步骤A3

## 阶段3：[模块B]（约1小时）
...
```

## 3.2 逐个找AI写代码

**不是把整个文档扔给AI让它一次干完。** 是每次只推进一个小块：

```
# 第1条消息
你好，我要开发一个跨境电商工作流。这是我的规划文档：[粘贴]
我们先开始阶段1：基础设施搭建。帮我完成项目文件夹结构和Playwright连接测试的代码。

# 第2条消息（阶段1验证通过后）
阶段1没问题。现在开始阶段2.1：写读取采集箱商品列表的代码。

# 第3条消息（阶段2.1完成后）
现在开始阶段2.2：...
```

**每完成一个小块，验证通过后才进入下一块。**

## 3.3 一个脚本的基本结构

当AI给你写代码时，你看到的Python文件会长这样：

```python
#!/usr/bin/env python3
"""这个文件是做什么的——（给以后看的人读）"""

# 导入需要用到的工具
import time
from playwright.sync_api import sync_playwright

# 真正的执行代码
def do_something():
    print("开始执行...")
    # ... 你的操作 ...
    print("完成")

# 入口（从这里开始执行）
if __name__ == "__main__":
    do_something()
```

> 💡 **小技巧：** 每个 `.py` 文件开头用 `"""..."""` 写清楚**这个文件是干啥的**。你以后绝对会感谢自己。

## 3.4 重要：管理和传递结果

AI写代码时，你要确保每个步骤的输出能传给下一步。

| 传递方式 | 适用场景 | 例子 |
|---------|---------|------|
| 命令行参数 | 简单的值 | `--store "順順の小屋童裝"` |
| JSON文件 | 复杂数据 | `{"pass_ids": ["id1","id2"]}` |
| 打印到屏幕 | 看执行过程 | `print("审查完成")` |

让AI写代码时明确告诉他：

```
每个步骤执行完后，把结果存到 result.json 文件里，格式：
{
    "success": true/false,
    "count": 数字,
    "ids": ["id1", "id2", ...]
}
```

# 第四篇：验收与Debug — 跑通才算完

## 4.1 普通操作：验证和容错

### 为什么需要验证？

**因为代码不会告诉你它做对了没有。** 脚本说"认领完成"，你真的去打开浏览器看，可能根本没认领成功。

### 每步操作后验证

```python
# 做了某项操作后...
page.click("text=认领")
time.sleep(3)

# 验证：读页面的文字确认
text = page.evaluate('document.body.innerText')
if "认领成功" in text:
    print("✅ 认领成功")
else:
    print("❌ 认领可能失败，需要人工检查")
```

### 加容错（不怕出错）

```python
import time

# 重试3次
for attempt in range(3):
    try:
        page.click("text=发布")
        break  # 成功了就跳出循环
    except:
        print(f"点击失败，第{attempt+1}次重试...")
        time.sleep(2)
```

### 加超时保护

```python
import subprocess

try:
    result = subprocess.run(
        ["python", "step2_review.py"],
        timeout=300,  # 最长等5分钟
    )
except subprocess.TimeoutExpired:
    print("❌ 超时了，这个步骤跑了超过5分钟")
```

**告诉AI你需要这些：**

```
帮我加上：
1. 每步操作后从页面读文字验证
2. 操作失败自动重试3次
3. 每个脚本设5分钟超时
```

## 4.2 AI验收：跑通才算完

### 验收清单

| 项目 | 怎么做 |
|--------|--------|
| ❌ 报错不跑 | 把错误信息复制给AI让它修 |
| ✅ 能跑但结果不对 | 检查逻辑——AI可能理解错了你的需求 |
| ✅ 能跑结果也对 | 恭喜，下一个模块！ |
| ⚠️ 看不懂AI写的 | 让它加注释，或让它解释逻辑 |

### 跑不动怎么办

把错误信息**完整**发给AI：

```
我跑了你给的代码，报了这个错误：

[把完整的报错信息贴在这里]

请帮我看看哪里出问题了。
```

AI 会分析错误原因并给出修复代码。

### 觉得不对怎么办

```
你写的代码逻辑我不确定对不对。我的需求是：

[把你的需求再说一遍，更清楚一点]

你现在的写法是 [描述你的理解]，对吗？
如果不对，应该怎么改？
```

## 4.3 Debug场景：常见问题让AI修

### 报错了
```
Traceback (most recent call last):
  File "test.py", line 15, in <module>
    from playwright.sync_api import sync_playwright
ModuleNotFoundError: No module named 'playwright'

→ 把完整的错误发给AI
→ AI回复：需要 pip install playwright
→ 你照做
```

### 逻辑不对
```
我跑了你写的审查代码，它能跑了，但我觉得审查结果不对。
所有商品都被拒绝了，但我手动看有一些是可以的。
帮我分析可能的原因。
```

### 性能太慢
```
合规审查每件商品要30秒，100件要50分钟，太慢了。
有没有办法优化，比如批量调用或并行处理？
```

### 不确定怎么改
```
我想添加一个功能：如果审查全部被拒绝，能不能自动采集一批新商品再跑一次？
帮我规划一下怎么改，不要直接写代码，先说说思路。
```

```
我 → AI：我想做一个跨境电商工作流，把采集箱商品自动审查→认领→发布到Shopee。
```

```
    提取所有商品的：货源ID、标题、价格、图片URL

```

```
  （知道每段代码大概在干什么）

  （基础逻辑自己写，复杂的让AI辅助）

  （你的核心竞争力是「知道要做什么」和「知道怎么验收」）
```

# 附录：常见问题 FAQ

### Q1：AI写的代码我完全看不懂怎么办？

**不需要完全看懂。** 你只需要知道：
- **输入是什么** → 你要给它什么数据
- **输出是什么** → 它给你什么结果
- **怎么运行** → `python xxx.py` 回车

### Q5：Windows 中文乱码怎么办？

看到这个错误：
```
UnicodeEncodeError: 'gbk' codec can't encode character
```

运行命令前加：
```bash
set PYTHONIOENCODING=utf-8
python run_all.py
```

或者告诉AI：`帮我加上GBK编码兼容处理`
